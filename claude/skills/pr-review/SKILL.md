# PR レビュー Skill

証拠のないレビューはしない。差分の見た目ではなく、実際のコードに照らして各指摘を
検証する。あなた（ユーザー）は permission-sensitive な操作（force-push、Slack 返信、
承認の投稿）を手元で握りたい人なので、投稿・承認は**必ず事前に内容を提示して確認を取る**。

## 前提の確定（最初に必ず）

1. **対象を曖昧なまま探し始めない**。repo / branch / PR 番号を確定する。
   「Slack のあれ」等と言われたら、まず Slack スレッド等の一次ソースを開いて対象 PR を特定してから着手する（ローカルを闇雲に grep しない）。
2. `git fetch` して remote 最新に同期する。**stale な状態で「fix が入っていない」と結論しない**。

## レビュー手順

1. `gh pr view` / `gh pr diff` で差分・説明・base ブランチを取得。
2. 既存の inline コメント・CodeRabbit / AI レビューコメントを取得し、対応済み/未対応を仕分け。
3. 各指摘を**実コードで検証**する。差分だけで判断せず、必要なら関連ファイルを**全体**読む
   （部分読みで結論しない）。自分の過去の提案が誤っていないかも疑う。
4. 指摘を重大度で分類:
   - **Critical**: バグ・セキュリティ・データ破壊・後方互換性の破壊
   - **Major**: 設計上の問題・抜けたエラーハンドリング・テスト不足
   - **Minor**: 可読性・命名・軽微な非効率
   - **Nit**: 好み・スタイル
   各指摘に「根拠（ファイル:行、理由）」を必ず添える。
5. 返信ドラフトと承認可否（approve / request changes / comment）を提示し、**投稿前に確認を取る**。

## 複数 PR の並列レビュー

3〜6 本のような複数 PR は、PR ごとに 1 サブエージェントを並列起動して捌く
（[[model-policy]] に従い sonnet 既定）。トップレベルはコーディネータとして:

- 各エージェントの結果を集約し、重複指摘を除く。
- **PR 横断の懸念**（共通のカップリング、stacked PR の依存順）を洗い出す。
- 何も投稿する前に、統合レポートを1つにまとめて提示する。

## 注意事項

- 「問題が無ければ無いと言う。粗探しで問題を捏造しない」。
- 承認・コメント投稿・thread resolve は破壊的ではないが outward-facing。必ず事前確認。
