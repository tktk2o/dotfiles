# 根本原因調査 Skill

推測で結論しない。実データ・実ログ・実コードで裏を取ってから原因を述べる。
このワークフローの最大の失敗は「部分的な情報から自信満々に間違える」こと
（例: 1カラムだけ見て判断、JOIN が黙って空を返す、ファイルの一部だけ読む、
stale な git 状態で「未適用」と誤認）。以下を**不変条件**として守る。

## 不変条件（結論前に必ず）

1. **鮮度**: コードの適用状態を語る前に `git fetch` し remote と照合する。
2. **全体読み**: コード/値を探すときは対象ファイルを**全部**読んでから答える。部分読みで結論しない。
3. **クエリ検証**: データ調査では、JOIN/フィルタが**期待したカラム・行数を返しているか**を確認する。
   単一フィールドだけで結論しない（判定に効く別フィールドの見落としを疑う）。
4. **前提の明示と反証**: 結論の前に、依拠している前提を列挙し、**代替仮説を最低1つ**証拠で潰す。

## 調査手順

1. **スコープ確定**: 対象（チケット / Slack スレッド / 事象）と再現条件を特定する。
2. **証拠収集**: 一次ソース（DB の実データ、ログ、メトリクス、該当コード）から証拠を集める。
   surface な答えで済ませない。
3. **原因特定**: 不変条件を満たした上で、根本原因を1つに絞る。
4. **影響定量化**: 「N 件中 M 件」のように影響範囲を数値で出す。
5. **提示**: 根拠（どのデータ/コードが何を示すか）と**確信度**を添えて報告する。
   確信度が低ければ、追加で確認すべき点を明示する。
6. **修正**: 証拠が決定的なときのみ、修正の Draft PR を用意する（本番反映はユーザー確認）。

## 長時間タスクの扱い

翻訳・深掘り調査・大規模分析などトークン/セッション上限に当たりやすい作業は、
**明示的なフェーズに分割**し、各フェーズ完了ごとに中間結果をファイルに保存してから次へ進む。
途中で切れても進捗を失わないようにする。
